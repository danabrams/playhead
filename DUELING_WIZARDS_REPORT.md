# Dueling Idea Wizards Report: Playhead

> 2026-04-29. Generated by orchestrating Claude Code (Opus 4.6) and OpenAI Codex (gpt-5.3-codex, xhigh) through study → ideate → cross-score → reveal → blind-spot phases.

---

## Executive Summary

Two AI models independently generated 30 ideas each for improving Playhead, then winnowed to 5, cross-scored adversarially (0–1000), reacted to each other's scoring, and finally generated 5 "blind spot" ideas after the adversarial round expanded their context. The duel produced unusually clean signal because the two models attacked the prompt from **structurally different angles**: CC pursued user-facing transcript-product features; COD pursued reliability/infrastructure hardening. Cross-model concession was honest on both sides — CC conceded that 3 of its 5 ideas were already shipping or already beaded; COD conceded that 2 of its 5 ideas duplicated existing infrastructure.

**Strong consensus winners (already-tracked beads, both models scored ≥700):** Cross-episode transcript search (`playhead-90i`) and intelligent sleep timer (`playhead-g21`) — both have full implementation specs and should be the next two execution targets.

**Most novel survivor not in any backlog (highest single-pick value):** Episode summaries via on-device FM — the only one of CC's ideas that doesn't overlap an existing bead and that COD scored at 755.

**Highest-leverage blind-spot idea (introduced post-adversarial):** Structure-aware silence compression using the existing FeatureWindow data — a "Smart Speed killer" that uses 14+ already-computed audio features to compress music beds and dead air without touching speech cadence.

## Methodology

- **Agents:** Claude Code (Opus 4.6, 1M ctx) at pane 0; OpenAI Codex (gpt-5.3-codex, reasoning_effort=xhigh) at pane 1. Gemini omitted at user request.
- **Project study:** Both agents read `AGENTS.md`, `CLAUDE.md`, key `PLAN.md` / `BEADS.md` / `PROGRESS.md` files, and toured the codebase (`Services/PreAnalysis/`, `PlayheadRuntime`, etc.) before producing ideas. CC finished study in 3m 30s; COD over-explored to 21m before being interrupted.
- **Ideation:** 30 candidate ideas → top 5, with full rationale per idea.
- **Cross-scoring:** Each agent scored the other's top 5 on a 0–1000 rubric (idea quality, real-world utility, implementation practicality, complexity-justified, design-identity fit).
- **Reveal:** Each agent saw the other's scoring of *their* ideas and wrote honest reactions — agreements, disagreements, and self-revisions.
- **Blind-spot probe:** With both agents now adversarially expanded, each generated 5 ideas neither side had originally proposed.

All 8 artifacts (`WIZARD_*.md`) are preserved alongside this report.

## Consensus Winners (≥700 from both sides)

### #1 — Cross-Episode Transcript Search

**Cross-model verdict:** CC self-ranked #2; COD scored 905 (highest of any idea). Already tracked as **`playhead-90i`** with full implementation spec (TranscriptSearchService API, SearchView with debounce + pagination, FTS5 highlight() rendering, 8 acceptance criteria).

**Why both agents agreed:** The infrastructure is already built — `transcript_chunks_fts` virtual table, content-sync triggers, `AnalysisStore.searchTranscripts(query:)` method. Hundreds of hours of searchable transcript text are currently locked behind per-episode views. Search transforms Playhead from a podcast player into a podcast knowledge base — a category-of-one capability no other app has.

**The contested point:** CC noted that COD's scoring methodology was inconsistent (penalizing pre-analysis for being "already implemented" while not penalizing search for being "already specified as a bead"). Fair critique, but the practical answer is: ship the bead. The work is scoped, the design is specified, and the value is unambiguous.

**Action:** Promote `playhead-90i` to next-up on the queue.

---

### #2 — Sleep Timer with Intelligent Stop Points

**Cross-model verdict:** CC self-ranked #5 (550), COD scored 870, CC self-revised upward to 700 after reading the existing bead spec. Already tracked as **`playhead-g21`** with detailed spec (5-state state machine, sentence-boundary search, 2s volume fade, ad-region awareness, 10 acceptance criteria).

**Why both agents agreed:** Uses already-computed `pauseProbability` per `FeatureWindow` to stop at natural break points instead of mid-sentence. The killer detail in the existing bead spec is **"if timer fires during a detected ad, seek to ad end first"** — a brilliant interaction between the sleep timer and ad detection that no other podcast app can match. The two Playhead-specific capabilities compound.

**The structural argument for shipping:** Sleep timers are one of the most-mentioned features in podcast app reviews. Making this a table-stakes feature *with* Playhead-specific differentiation is high-leverage daily-use polish.

**Action:** Promote `playhead-g21` after `playhead-90i`.

## Strongest Novel Idea (Not Already Tracked)

### Episode Summaries via On-Device Foundation Model

**Origin:** CC #3. **Cross-score:** 755 (COD). **Bead status:** Not tracked.

**Why this matters:** This is the only idea in either agent's top-5 that **isn't already in the backlog**. Of the original ten top-5 ideas across both agents, three are already shipping/built and four are already beaded. Of the remaining three (CC #3, CC #4, COD #1), this scored highest cross-model.

**The mechanism:** After backfill transcription reaches sufficient coverage (e.g., ≥80%), queue a summary extraction job. Sample representative transcript text (first 3 min + middle segment + last 3 min — avoids feeding 50K tokens). Feed to on-device FM via schema-bound `@Generable` struct (`summary: String`, `mainTopics: [String]`, `notableGuests: [String]`). Store in new `episode_summaries` table. Surface in EpisodeListView as expandable subtitle, topic tags as small Soft Steel pills.

**COD's most useful concern:** "Hallucination/over-summary risk can erode trust fast" — needs guardrails: confidence gating, explicit fallback copy, very restrained UI placement, robust regression tests on known-episode fixtures.

**CC's defense:** The FoundationModelClassifier infrastructure (schema-bound generation + capability gating + permissive content path for guardrail refusals) is proven in production. Falls back gracefully to standard show-notes when FM unavailable.

**Action:** Open a new bead. This is the highest-value non-beaded idea from either agent.

## High Value, Not Yet Beaded

### Shareable Transcript Quotes (CC #4, COD score 710)

**Why both agents agreed:** Tiny implementation (~200 lines SwiftUI + `QuoteFormatter`), uses standard `UIActivityViewController`, no new services. Editorial format ("...quote text... — Show Title, Episode, Timestamp, Shared from Playhead").

**The asymmetric point CC made that COD missed:** This is the **only idea in either list that creates user acquisition**. Every other idea improves the experience for existing users. Shareable quotes are the feature where someone who doesn't use Playhead sees a beautifully formatted podcast quote in social media, asks "what app made that?", and downloads. For a pre-launch app, that's existential.

**COD's caveat:** Risk of drifting toward "growth-hacky branding chrome." Mitigated by editorial typographic format, no emoji, understated "Shared from Playhead" attribution.

**Action:** Open a bead at P2 (smaller scope, defer behind search + summaries + sleep timer).

**Action:** Don't open a new bead for "predictive pre-analysis." Instead, audit the existing pre-analysis pipeline for coverage-quality gaps and ship a quiet readiness affordance (a single subtle indicator on the episode cell, **not** a stage timeline or ETA).

## Killed Ideas (Mutual Agreement They're Weak)

### Unified Resource Governor (COD #2)
- **Score:** 250 (CC) / self-revised 220 (COD).
- **Cause of death:** `QualityProfile` + `AdmissionGate` + `DeviceClassProfile` + `StorageBudget` already implement exactly this consolidation. The Phase 3 memory documents that `DeviceAdmissionPolicy` was specifically removed and `QualityProfile` absorbed its role. **COD explicitly conceded duplication.**
- **Lesson:** Always check `QualityProfile.swift` header comment before proposing resource-management ideas.

### Readiness Timeline + ETA in Activity (COD #4)
- **Score:** 300 (CC) / self-revised 260 (COD).
- **Cause of death:** Stage names + ETA = exactly the metrics dashboard the "peace of mind, not metrics" feedback memory rejects. ETAs on on-device ML are unreliable (thermal/contention variance) — setting expectations that break is worse than no expectation. **Both agents agreed in reveal.**
- **What might survive a redesign:** A binary "ready" affordance (subtle mark on the episode cell when pre-analysis is complete) — borrowed from CC's salvage of pre-analysis readiness. **Not** a timeline, **not** an ETA.

### Skip Posture Dial (half of COD #5)
- **Score:** 350 (CC) / self-revised 390 (COD, after split-out).
- **Cause of death:** Per-show trust system (`PodcastProfile.mode` + `skipTrustScore` + `observationCount`) already adapts per show based on observed precision. Adding Strict/Balanced/Aggressive presets creates interaction conflicts ("user said Aggressive, trust system says Manual — who wins?") and conflates *behavior* with *confidence*.
- **What survives separately:** The "Why Skipped" evidence panel already exists as `AdRegionPopover` — making it more discoverable from the timeline (rather than only via transcript-peek tap) is genuine but minor UI polish, not a top-5 idea.

## Meta-Analysis: What the Disagreement Revealed

**CC bias profile:**
- Pursues **user-facing transcript-product features** by default.
- Critically benchmarks every claim against the actual codebase (caught 5 instances where COD proposed already-built infrastructure).
- Strong product instinct (correctly identified that quotes are the only acquisition lever, that ETAs fight identity, that posture presets conflate behavior with confidence).
- Weak point: doesn't always check the bead database before claiming novelty (3 of its top 5 turned out to be either shipping or already-spec'd beads). Self-prescribed fix: "run `bd list --limit 0 | grep <keyword>` before writing each candidate up."

**COD bias profile:**
- Pursues **reliability/control-plane infrastructure** by default. Four of its top 5 are pipeline-watchdog/governor/scheduler ideas; only one is user-facing.
- More generous in scoring (avg 772 on CC's ideas vs CC's avg 374 on COD's); under-bench-checks novelty claims.
- Strong engineering instinct on observability and stage-level recovery.
- Weak point: did not recognize that `QualityProfile` had already absorbed the role of the "Unified Resource Governor" it proposed — proposing infrastructure that already exists. Self-prescribed fix: scope reliability work as incremental enhancements to existing primitives (reconciler, lease, cause taxonomy) rather than new orchestration layers.

**Where the duel produced its best output:**
1. **Reveal-phase concessions.** Both agents made specific, evidence-backed concessions when shown the other's scoring. CC: "I was describing a feature that already ships." COD: "I would now score [Resource Governor] much lower as originally stated, because it duplicates existing architecture patterns." These are higher-signal than either's first pass.
2. **The shared blind spot diagnosis (CC).** Both lists were biased toward "build new thing" rather than "use existing thing differently." The strongest blind-spot ideas (B1 silence compression, B3 fingerprint transfer) come from asking "what data is Playhead already computing that it isn't using?" — a different question than "what feature should we add?"
3. **Convergence on Hands-Free / CarPlay / Route-awareness** without cross-talk. CC volunteered Siri/CarPlay/Widgets in reactions; COD independently produced App Intents + CarPlay in blindspots. Both hit route-aware adaptation. Strong independent convergence = strong signal.

## Recommended Next Steps

In priority order, derived from cross-model consensus, novelty against the backlog, and design-identity fit:

1. **Ship `playhead-90i` (Cross-episode transcript search UI).** Strongest consensus, infrastructure ready, full spec exists.
2. **Ship `playhead-g21` (Sleep timer with transcript-aware pause).** Strong consensus, existing data + ducking mechanism, full spec exists.
3. **Open a bead for Episode Summaries via On-Device FM (CC #3).** Highest-value novel idea (no existing bead). Use FoundationModelClassifier pattern. Add specific guardrails per COD's reaction: confidence gating, fallback copy, restrained placement, regression tests on known episodes.
4. **Open a bead for Structure-Aware Silence Compression (Blind Spot B1).** Highest-scoring blind-spot idea. Uses already-computed FeatureWindow data. Scoped to ~300 lines. Test carefully on AirPods/speakers/car audio before promoting.
5. **Open a bead for Shareable Transcript Quotes (CC #4).** Smallest implementation, only acquisition-lever idea in either list, fits Quiet Instrument aesthetic with editorial format.
6. **Audit pre-analysis readiness UX (salvage from CC #1 contested).** Don't rebuild — verify coverage-quality gaps exist, ship a single subtle "ready" affordance on the episode cell. **Not** a timeline. **Not** an ETA.
7. **Defer:** Pipeline Watchdog (frame as reconciler enhancement, ship only after observed in-session stall); Adaptive Hot-Zone Sizing (real but narrow benefit, simpler tier-based version is enough); Hands-Free Intents + CarPlay surface (both agents converged — open a single bead grouping route-aware behaviors at P3).
