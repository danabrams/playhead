## Why this doc

The `frozen-trace-v3` bump landed in #123 (q45f.1, listen-rewind capture). It adds `listenRewindEvents` to FrozenTrace and is consumed by the q45f counterfactual gate (#124). The previous baseline writeup (`2026-04-24-real-data-findings.md`) predated that bump. This doc locks in the rollup numbers under the v3 struct so future regressions are diffable against a stable reference.

## v3-specific observations

1. **No metric movement from the v3 schema bump itself.** All 128 fixtures decode `listenRewindEvents: []`. The q45f counterfactual gate processes those events; with empty arrays it makes no rollup contribution. ALL/Conan/DoaC numbers are identical to what would result under v2 decoding.

2. **Listen-rewind signal is unobserved in the corpus.** No fixture was captured with the v3 capture path active (none stamp `traceVersion: "frozen-trace-v3"` on disk). The q45f gate's behavior under non-empty rewind events is currently exercised only by unit tests, not by replay.

3. **Stamp-vs-recapture is a deliberate choice.** Mass-stamping existing v2 fixtures to v3 with `listenRewindEvents: []` would lie about capture-time semantics: per FrozenTrace docstring, a v3-stamped fixture promises the listen-rewind capture path was active, so an empty array means "no rewinds" rather than "field not captured". Stamping conflates those.

## Recommendation

A device-capture round (option C) is **only worth the time when there's a q45f-related signal hypothesis to validate**. Today, the q45f counterfactual gate has no behavioral coverage from the corpus — but no gtt is currently asking it a question. Defer fresh capture until either (a) the next q45f tuning round, or (b) a user-visible false-positive incident pulls listen-rewind sequences into the discussion.

The current corpus is a sound baseline for non-q45f work.
