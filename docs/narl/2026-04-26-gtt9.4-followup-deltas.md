# NARL gtt9.4 follow-up — sqhj deltas (2026-04-26)

This bead, `playhead-sqhj`, lands two changes flagged by `gtt9.4`'s spike:

1. **Wire `MusicBedLevel` into `buildAcousticLedgerEntries`** so the
   `.acoustic` source can fire on sustained music-bed coverage even when
   the boundary RMS-drop score is zero (was the cause of acoustic firing
   on only ~2 % of decisions in the 2026-04-23 dogfood capture).
2. **Flip `MetadataActivationConfig.default.counterfactualGateOpen` from
   `false` to `true`.** Per-gate flags (`lexicalInjectionEnabled`,
   `classifierPriorShiftEnabled`, `fmSchedulingEnabled`) all stay `false`
   — production behaviour is unchanged until a future bead also flips a
   per-gate flag. The flip removes the master kill that was hiding all
   downstream NARL knobs.

## What this doc captures

The original 2026-04-23 source-firing baseline (`acoustic 2 %`, `metadata
7 %`, `fm 5 %`, `lexical 3 %`, `catalog 3 %`, `classifier 100 %` of 147
windows) was computed from the dogfood-app `decision-log.jsonl` in
`.captures/2026-04-23/com.playhead.app 2026-04-23 13:54.43.748.xcappdata/`.
That log was emitted **by the build that was on-device that day**.
Reproducing it post-fix requires deploying a new dogfood build to a real
device and replaying the same 16 episodes — which is not reasonably
automatable from this worktree.

What **is** automatable is the NARL counterfactual harness against the
17 `FrozenTrace-*.json` fixtures derived from that capture. Those
fixtures encode the *recorded* evidence catalog as it stood on 2026-04-23
— they don't re-run `buildAcousticLedgerEntries`, so the acoustic-source
percentage in the fixtures doesn't move just because we fixed the
acoustic ledger code.

The honest report below: harness pass status post-sqhj, plus the harness
metrics it does compute on the 2026-04-23 cohort, plus a clear flag of
what still requires a device re-capture.

## Acoustic-source firing — what's testable now

The fixtures-driven harness can't re-derive "% of windows where
`.acoustic` fires" because the FrozenTrace evidenceCatalog is frozen.
What we **can** assert directly is the new code's behaviour on synthetic
inputs, which is what `AcousticLedgerMusicBedTests` (added in commit
`16b9ef7`) pins:

- **Music-bed coverage with zero RMS drop now fires `.acoustic`**
  (regression: pre-sqhj returned `[]`).
- **RMS drop alone still fires** (back-compat).
- **Below-30 % music coverage with no RMS drop does NOT fire** (matches
  `MusicBedLedgerEvaluator.minPresenceFraction`).
- **No signal of either kind** still returns `[]` (no zero-weight
  clutter).

Five tests, all green. The threshold values reused from
`MusicBedLedgerEvaluator` (`minPresenceFraction=0.30`,
`minWindowsRequired=3`) so no new tuning knobs were introduced.

## Source-firing measurement — re-capture path

To get post-sqhj source-firing percentages comparable to the
2026-04-23 baseline:

1. Build the dogfood scheme onto a paired device.
2. Replay the same 16 episodes that produced
   `.captures/2026-04-23/...13:54.43.748.xcappdata/`.
3. Pull the fresh `decision-log.jsonl` and recompute per-source counts
   the same way `2026-04-23-real-data-findings.md`'s "Finding 4" did.

Expected directional change (from the fix mechanics, not measured):

- `.acoustic` should rise above 2 % — every span with sustained music
  bed (≥30 % windows non-`.none`) now produces an entry.
- `.metadata` percentage is unchanged by sqhj — the master gate flip
  doesn't activate metadata cues for the lexicon (per-gate flags still
  off).

The size of the acoustic delta depends on how many spans actually carry
≥30 % `MusicBedLevel != .none` windows in the corpus, which we don't
have a precomputed answer for. A new device capture on the sqhj build
would tell us.

## What changed in this bead

- `Playhead/Services/AdDetection/AdDetectionService.swift`: augmented
  `buildAcousticLedgerEntries` with `musicBedPresenceFraction`; combined
  RMS-drop and music-bed coverage via `max()` into a single `.acoustic`
  entry (parallel `.musicBed` source path via
  `MusicBedLedgerEvaluator` is unaffected).
- `Playhead/Services/AdDetection/MetadataActivationConfig.swift`:
  `counterfactualGateOpen: false → true` in `.default`.
- Tests: new `AcousticLedgerMusicBedTests`, plus pinning tests in
  `MetadataActivationTests` for the gate-default flip; updated
  `DecisionLoggerTests` assertions to match.

## Unblocked work

With the master gate open by default, the priorShift-band retune
(gtt9.3, already shipped) and any future per-gate tuning beads can
actually land effects on the corpus. Before sqhj, flipping
`classifierPriorShiftEnabled` alone was a no-op because
`isClassifierPriorShiftActive` short-circuited on the closed master.
