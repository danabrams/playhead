# gtt9.1: Acoustic-likelihood-driven transcript scheduling — Shadow design

Date: 2026-04-27
Bead: `playhead-gtt9.1`
Status: shadow-mode landed; production skip remains off pending eval

## Problem statement

The 2026-04-26 dogfood capture (a few hours of listening) shows
**105,839 decision-log entries against 147 fusion-scored windows
(0.14%)**. The bottleneck is `BackfillEvidenceFusion
.quorumGateForFMConsensus` (`Playhead/Services/AdDetection/
BackfillEvidenceFusion.swift:479-495`), which requires
`distinctKinds.count >= 2` — most candidate windows ship a single
evidence kind (FM verdict OR acoustic break) and **no transcript**, so
the quorum gate drops them.

Transcript is also the dominant CPU/battery cost in the analysis
pipeline. Today every shard in `[0, desiredCoverageSec]` is handed to
`TranscriptEngineService.startTranscription` unconditionally
(`AnalysisJobRunner.swift:264-270`, original line 238). There is no
per-shard acoustic prefilter — the scheduler decision is made entirely
at job-enqueue time by `desiredCoverageSec`, not by per-window acoustic
evidence.

The hypothesis: gating transcript on per-shard acoustic likelihood
**simultaneously increases accuracy** (more borderline windows reach
2-kind quorum because the transcript budget is concentrated on
ad-likely regions) **and reduces processing time** (we stop transcribing
quiet host conversation that contributes nothing to the detector).

## Shadow-mode design

### What's shipping in this bead

The gate is fully implemented but **shadow-only**: the runner scores
every shard, logs the would-skip decision, but continues to transcribe
every shard. We collect ≥1 dogfood capture's worth of telemetry, then
decide whether to flip `skipEnabled` true.

Three new files:

* `Playhead/Services/AnalysisJobRunner/AcousticTranscriptGateConfig.swift`
  — master/per-stage gate config mirroring `MetadataActivationConfig`'s
  shape. Defaults: `enabled = true, skipEnabled = false,
  likelihoodThreshold = 0.30, safetySampleFraction = 0.10`.
* `Playhead/Services/AnalysisJobRunner/TranscriptShadowGateLogger.swift`
  — `TranscriptShadowGateLogging` protocol + `TranscriptShadowGateEntry`
  Codable record + `NoOpTranscriptShadowGateLogger` (production
  default).
* `PlayheadTests/Services/AnalysisJobRunner/AcousticTranscriptGateTests.swift`
  — seven Swift Testing scenarios exercising every decision arm.

One modified file: `Playhead/Services/AnalysisJobRunner/AnalysisJobRunner.swift`
— adds the `evaluateAcousticTranscriptGate` helper called between
feature extraction and transcription. The helper reuses the existing
`AcousticLikelihoodScorer.scoreOne` math via a new
`maxLikelihoodInSpan` reducer (added in
`Playhead/Services/PreAnalysis/AcousticLikelihoodScorer.swift`) that
takes the max scoreOne over feature windows overlapping a shard's
half-open span.

### Decision categories logged per shard

| Decision | When | `wouldGate` | `transcribed` (shadow) | `transcribed` (prod-skip) |
| --- | --- | --- | --- | --- |
| `aboveThreshold` | likelihood ≥ threshold | false | true | true |
| `wouldSkip` | likelihood < threshold, sample arm tails | true | true | false |
| `safetySampleKeep` | likelihood < threshold, sample arm heads | true | true | true |
| `qualityPreconditionKeep` | shard end ≤ asset's `fastTranscriptCoverageEndTime` | false | true | true |
| `scoreUnknown` | no `feature_windows` overlap the shard | false | true | true |

`wouldGate` is `true` exactly when production-skip mode would withhold
the shard — that's the column eval consumers will join on.

### Mitigations preserved

* **M1 — quality precondition**: when the asset's
  `fastTranscriptCoverageEndTime` already covers a shard, the runner
  bypasses scoring and tags the shard `qualityPreconditionKeep`. The
  gate never withdraws a shard mid-stream from an episode whose
  transcript downstream consumers are already depending on.
* **M2 — safety sample**: the default 10% safety-sample fraction
  re-tags ~10% of `wouldSkip` shards as `safetySampleKeep` so we keep
  a calibration stream of low-likelihood ground truth even after
  `skipEnabled` flips. This keeps the `skipEnabled = true` PR from
  shipping a never-exercised sampling code path.
* **M3 — shadow-mode first**: the shipped default
  (`enabled=true, skipEnabled=false`) writes the rows but never affects
  which shards reach the engine. Production behavior is byte-identical
  to the pre-gtt9.1 runner.

### Master gate pattern

`AcousticTranscriptGateConfig` mirrors `MetadataActivationConfig`:

* `enabled` — master kill. When false, the runner short-circuits
  before scoring and logging. No allocations, no sink calls.
* `skipEnabled` — production-skip switch. AND'd with `enabled` via
  `isProductionSkipActive`.
* `is*Active` computed properties read through the AND so every
  consumer sees one consistent surface.

This is the same shape `MetadataActivationConfig.counterfactualGateOpen
&& classifierPriorShiftEnabled` uses (see `MetadataActivationConfig
.swift:111-124`).

## Eval procedure

The shadow rows are emitted via `TranscriptShadowGateLogging`. The
production sink is currently `NoOpTranscriptShadowGateLogger` — a
follow-up wiring bead will install a JSONL writer that lands the rows
under `Documents/transcript-shadow-gate.jsonl` alongside
`decision-log.jsonl`. (Decoupling the sink keeps the hot path
shippable without committing to a particular file format yet.)

Once the dogfood JSONL writer ships, the eval procedure on a capture
is:

1. Pull the JSONL stream from a dogfood device
   (`transcript-shadow-gate*.jsonl`) and parse into a typed table.
2. Join each row's `(analysisAssetID, shardStart, shardEnd)` against
   the same capture's `decision-log.jsonl` so every shard has both
   the gate decision *and* the downstream classifier verdicts that
   landed inside the shard.
3. Compute the **would-skip rate**: rows where
   `decision == .wouldSkip` divided by all gate-eligible rows
   (i.e. excluding `qualityPreconditionKeep` and `scoreUnknown`).
   This sets the upper bound on transcript-time savings.
4. Compute the **host-read miss rate** on the would-skip rows: of the
   `wouldSkip` rows whose span overlaps a known host-read ad
   (manually labelled or recovered from `decision-log.jsonl` as
   `action == .autoSkipEligible` with `evidence.lexical` matched
   categories), how many would have been silently dropped. This is
   the figure that determines whether `skipEnabled` is safe.
5. Cross-reference the **fusion-window ratio** before/after a shadow
   replay: re-run the corpus with `skipEnabled=true` against the
   replay tooling and confirm the 0.14% → ≥5% target from the bead's
   acceptance criteria.

## Decision criteria for flipping `skipEnabled = true`

The follow-up bead may flip the production-skip switch when ALL of
the following hold against the shadow-eval corpus:

1. **Host-read recall ≥ 95% of pre-gate baseline** — at most 5% of
   labelled host-read ads land in `wouldSkip` shards.
2. **Fusion-window ratio ≥ 5%** in a `skipEnabled=true` replay (vs
   the 0.14% baseline) — the bottleneck the bead targets is
   actually relieved.
3. **No regression in held-out detection recall** — the existing
   counterfactual eval pipeline reports detection recall within the
   same 5% guard.
4. **Safety-sample rows present** in a sufficient ratio of would-skip
   shards (~10% by default config) — confirms the calibration stream
   for ongoing post-flip measurement is intact.

If any of these fail, the alternatives are: tune
`likelihoodThreshold` lower (more shards transcribe), raise
`safetySampleFraction` (more keeps), or extend
`AcousticLikelihoodScorer.Weights` to include a host-read-friendly
signal (e.g. lexical pre-warming from podcast metadata) before
revisiting the flip.

## What's explicitly out of scope

* The JSONL writer for `TranscriptShadowGateLogging` — the protocol
  ships with a no-op default and the structured entry type; the
  on-disk shape can be revised before the writer lands.
* Tuning `likelihoodThreshold` and `safetySampleFraction` from real
  data — the defaults are deliberate priors. Tuning is an eval-
  driven follow-up.
* Coupling to `gtt9.4` / `sqhj` (recently merged metadata-activation
  beads) — the gate operates on shard scheduling, not fusion logic,
  and runs strictly before the metadata path.
* Bypassing the `quorumGateForFMConsensus` `distinctKinds.count >= 2`
  check directly. The gtt9.1 thesis is that the quorum gate is
  correct; the diet of windows reaching it is the problem.
